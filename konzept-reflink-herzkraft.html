Herzkraft-Analyse

Empfehlungsprogramm für die Herzkraft-Analyse Landing Page — sauberes Tracking, Quentn-Integration und automatisierte Zuordnung.

Projekt: ImpactFlow Datum: 3. März 2026 Version: 1.1
1

Übersicht & Ziel

Partner (Coaches, Therapeutinnen, Netzwerk-Kontakte) sollen die Herzkraft-Analyse mit ihrem eigenen Link bewerben können. Jede Anforderung wird dem Empfehler zugeordnet.

🎯

Ziel

Jeder Partner erhält einen persönlichen Link. Wer über diesen Link die Analyse anfordert, wird automatisch dem Partner zugeordnet.

🔗

Link-Format

go.myimpactflow.ch/herz-analyse.html ?ref=partner-code&src=linkedin

📈

Tracking

Pageview, Formular-Absendung und E-Mail werden mit Partner + Plattform erfasst und in N8N gespeichert.

📧

Quentn-Integration

Der Empfehler-Code wird als Hidden Field direkt an Quentn übermittelt — sichtbar im Kontakt-Profil.

💡
Wichtig: Das System läuft komplett client-seitig (GitHub Pages). Kein Backend nötig. Tracking erfolgt via sendBeacon an N8N-Webhooks. Die Partner-Zuordnung geschieht über einen einfachen URL-Parameter.

IST-Zustand vs. SOLL

Komponente IST (heute) SOLL (nach Umsetzung)
RefLink-Generator Nur für WhatsApp-Seite (revlink.html) Erweitert für Herzkraft-Analyse

Komponenten im Detail

A — RefLink-Generator (Neue Seite oder Erweiterung)

Der Partner gibt seinen Namen oder Code ein und wählt die Plattform(en). Der Generator erzeugt den fertigen Link + QR-Code + PDF zum Ausdrucken.

Eingabe Beispiel Ergebnis
Partner-Name susanne-meier herz-analyse.html?ref=susanne-meier
+ Plattform LinkedIn herz-analyse.html?ref=susanne-meier&src=linkedin
+ Plattform Instagram herz-analyse.html?ref=susanne-meier&src=instagram
📄
PDF-Export für Partner: Enthält alle Links, QR-Code und Kurzanleitung — genau wie beim bestehenden revlink.html, aber für die Herzkraft-Analyse.

B — Landing Page (herz-analyse.html)

Was auf der Seite angepasst werden muss:

1. Parameter lesen

?ref= und ?src= aus der URL auslesen und in localStorage speichern.

2. Hidden Field setzen

Neues Feld field_empfohlen_von im Formular mit dem Partner-Code befüllen.

3. sendBeacon erweitern

Bei Formular-Submit den ref-Code zusätzlich an den N8N-Tracking-Webhook senden.

4. Pageview-Tracking

Beim Seitenaufruf mit ?ref= ein Event an N8N senden (Besuch zählen).

Datenfluss im Formular:

// Beim Laden der Seite:
var ref = params.get('ref');   // z.B. "susanne-meier"
var src = params.get('src');   // z.B. "linkedin"
localStorage.setItem('if_ref', ref);
localStorage.setItem('if_source', src);

// Hidden Field im Formular befüllen:
document.getElementById('ref-field').value = ref;

// Bei Submit → Quentn erhält automatisch:
// field_empfohlen_von = "susanne-meier"

Quentn & N8N Integration

C — Quentn: Custom Field anlegen

In Quentn muss ein neues benutzerdefiniertes Feld angelegt werden, damit der Empfehler-Code direkt am Kontakt gespeichert wird.

Einstellung Wert
FeldnameEmpfohlen von
Technischer Namefield_empfohlen_von
TypTextfeld (einzeilig)
Sichtbar inKontaktdetails
⚠️
Manueller Schritt in Quentn: Das Custom Field muss in der Quentn-Oberfläche unter Kontakte → Felder verwalten angelegt werden. Der technische Name wird von Quentn automatisch generiert — diesen brauchen wir dann für das Hidden Field im Formular.

D — N8N: Zwei Workflows

⚙️

Workflow 1: RefLink Conversion Tracker

Webhook: /webhook/reflink-herz-conversion
Empfängt: { ref, src, email, variant, timestamp }
Aktionen: DataTable speichern, Slack-Nachricht, optional Quentn-Suche

📋

Workflow 2: Wochen-Report

Trigger: Schedule (Freitags 17:00) + Slack-Button
Aktionen: DataTable auslesen, nach Partner + Plattform gruppieren, AI-Zusammenfassung, Slack-Report

N8N DataTable Struktur:
Spalte Typ Beispiel Beschreibung
refTextsusanne-meierPartner-Code

srcTextlinkedinPlattform
variantTextBA/B-Test Variante
eventTextconversionpageview oder conversion
timestampDateTime2026-03-02T14:23:00ZZeitpunkt

Schritt-für-Schritt Implementierung — von der Vorbereitung bis zum Go-Live.

Phase 4 — N8N Workflows
Conversion Tracker + DataTable

End-to-End testen + Ausrollen
Test-Partner-Link erstellen → Analyse anfordern → Prüfen: Quentn-Feld, DataTable-Eintrag, Slack-Meldung. Dann Push + ersten echten Partnern den Generator geben. Go-Live

Geschätzter Aufwand: Phase 1 (5 Min.) + Phase 2 (30 Min.) + Phase 3 (45 Min.) + Phase 4 (60 Min.) + Phase 5 (20 Min.) = ca. 2–3 Stunden Gesamtaufwand.

Zukünftige Verbesserungen

Geplante Optimierungen für Landing Page, Conversion und Technik — priorisiert nach Wirkung.

Glaubwürdigkeit & Konsistenz Prio 1

  • IP-basierter Counter-Schutz (Backend) N8N-Webhook der Hochstwert pro IP speichert. Zähler sinkt nie — auch bei Inkognito-Modus oder anderem Gerät. Aktuell nur localStorage (pro Browser).
  • Counter-Wachstum begrenzen Bei vielen Page-Reloads steigt der Besucher-Counter unrealistisch schnell. Tageslimit oder zeitbasierte Wachstumsrate einbauen.
  • Verhältnis Besucher ↔ Analysen validieren Logische Konsistenz prüfen: Wenn 28 Besucher online, aber nur 8 Analysen heute — passt das? Automatische Verhältnis-Korrektur.

Conversion-Optimierung Prio 2

  • A/B-Test Auswertung Variante A vs. B: Welche konvertiert besser? Tracking-Daten aus Quentn und N8N analysieren. Gewinner-Variante ausrollen.
  • Danke-Seite erweitern Sequenzielles Pulsieren (Step 1→2→3), Spam-Hinweis inline unter Step 1, erledigte Steps gedimmt mit ✓. Offen: Social Sharing, Empfehlungs-Link, Countdown.
  • Exit-Intent Popup Besucherinnen die die Seite verlassen wollen nochmal ansprechen. Sanfter Reminder mit Kern-Nutzenversprechen.
  • Formular-Abbruch-Tracking Wo genau brechen User im 2-Step-Formular ab? Welche Frage hat die höchste Absprungrate? Event-Tracking pro Schritt.

Technik & Performance Prio 3

  • Cache-Busting Meta-Tags (no-cache, no-store, must-revalidate) auf herz-analyse.html + danke.html. Browser fragt bei jedem Besuch beim Server nach.
  • Lighthouse / Core Web Vitals Performance-Score prüfen und optimieren. Bilder komprimieren, CSS/JS minimieren, LCP und CLS verbessern.
  • Error Monitoring JavaScript-Fehler auf der Live-Seite automatisch tracken. window.onerror → N8N-Webhook → Slack-Alert bei Fehlern.
  • Script-Block aufteilen Aktuell ein einziger <script>-Block (3000+ Zeilen). Ein Fehler in einer Funktion killt alles. Aufteilen in separate Blöcke oder try/catch pro Feature.
